## Загальна оцінка (консенсус)

**Аналітичні запити (заперечення, скрипти, оцінка) — 6/10.** Бот слідує ШАБЛОНУ, але НЕ ДУМАЄ. Кожна SOS-відповідь виглядає однаково незалежно від контексту. Менеджер отримує "правильну структуру з generic наповненням" замість "глибокий аналіз саме цієї ситуації".

## 1. ПРОМПТ-ІНЖЕНЕР

### Діагноз: GPT виконує FORMAT, але не виконує LOGIC

**Проблема 1: SOS — один шаблон на все**
Тест SOS-1 (ціна) і SOS-2 (конкурент) дають майже ідентичну структуру:
```
Крок 1: питання
Крок 2: гілки (конкурент/ціна/сумнів/лояльність) — ВСІ 4 гілки кожного разу
Суть: фінансовий/технічний/репутаційний
Killer phrase
```
GPT генерує ВСІ гілки замість тієї однієї, яка релевантна. Якщо заперечення = ціна, навіщо гілка "лоялен до конкурента"?

**Рішення:** Змінити формат з "покажи всі гілки" на "обери ОДНУ гілку і розкрий глибоко":
```
💬 Крок 2 — ОБРАНА СТРАТЕГІЯ (одна з 4, найрелевантніша):
[тип]: «[розгорнута відповідь з 2-3 фактами]»
Чому ця стратегія: [1 рядок — чому саме вона для цього заперечення]
```

**Проблема 2: SPIN-ланцюжок ігнорується**
У жодній з 2 SOS-відповідей SPIN не з'явився. GPT бачить SPIN в кінці промпту і пропускає — priority bias (те що на початку важливіше для моделі).

**Рішення:** Перенести SPIN вище — одразу після Крок 1, перед Суть. Або зробити SPIN частиною Крок 1:
```
💬 Крок 1 — SPIN-відкриття (менеджер ПИТАЄ, не розповідає):
S: «[ситуаційне питання]»
→ якщо лікар відповів — перейди до:
P: «[проблемне питання]»
```

**Проблема 3: Killer phrase — generic**
Тести показують killer phrases типу "Neuramis пропонує вищу маржу при порівнянній якості" — це 12 слів, але це не killer phrase. Killer phrase = те що менеджер ДОСЛІВНО скаже лікарю. Поточні — це summary, а не фраза.

**Рішення:** Додати few-shot приклади:
```
🔑 *Killer phrase* — фраза яку менеджер скаже лікарю ДОСЛІВНО:
Приклади рівня якості:
✅ «Один шприц Ellanse замінює 3 процедури Sculptra — порахуйте час»
✅ «Ваша маржа з Neuramis на 40% вища ніж з Juvederm — при тій самій якості»
❌ «Ellanse має багато переваг для вашої практики» — GENERIC, переписати
```

**Проблема 4: Extract працює, але GPT не завжди ВИКОРИСТОВУЄ extracted facts**
У SOS-1 extract витягнув "маржа 4500-11500 грн", але GPT в Суть написав "забезпечує вищу маржу" без цифри. Факт-форсинг в BASE є, але GPT ставить його на 2-й пріоритет після формату.

**Рішення:** В кожному sub-prompt додати micro-reminder:
```
⛔ ПЕРЕД відправкою: перевір що КОЖЕН пункт містить цифру з ВИТЯГНУТИХ ФАКТІВ.
Якщо є "забезпечує вищу маржу" без цифри — ЗАМІНИ на "маржа 4500-11500 грн за процедуру"
```

## 2. ТРЕНЕР З ПРОДАЖІВ

### Діагноз: Бот дає ПРАВИЛЬНЕ, але не дає ЕФЕКТИВНЕ

**Проблема 5: Відсутня адаптація під тип лікаря**
Всі відповіді написані для "абстрактного лікаря". Реальні менеджери працюють з:
- Досвідчений косметолог (15+ років) — не терпить базових пояснень
- Молодий дерматолог — потребує впевненості і фактів
- Пластичний хірург — мислить процедурами, не продуктами
- KOL (Key Opinion Leader) — потребує наукові аргументи

Бот не питає і не адаптує. Відповідь для KOL і для джуніора однакова.

**Рішення:** Додати в SOS і SCRIPT блок адаптації:
```
🎯 АДАПТАЦІЯ (визнач з контексту діалогу):
- Досвідчений лікар → коротко, без базових пояснень, тільки цифри і диференціація
- Молодий лікар → впевнений тон, step-by-step, не перевантажуй
- Скептик → тільки факти + fear-based, без емоцій
Якщо тип невідомий — дай версію для досвідченого (краще коротше ніж довше)
```

**Проблема 6: Немає "антимонолог" правила**
Тест SCRIPT-1 показує скрипт де менеджер говорить 4 речення поспіль. Це МОНОЛОГ. Лікар відключиться після 2 речення.

**Рішення:** Жорстке правило для SCRIPT:
```
⛔ ПРАВИЛО АНТИМОНОЛОГ:
Менеджер НІКОЛИ не говорить більше 2 речень поспіль.
Після кожних 2 речень — ПИТАННЯ до лікаря.
Скрипт = діалог, не презентація.
```

**Проблема 7: "Покращена версія" в EVALUATE — теж шаблонна**
Тест EVALUATE-1 (оцінка 6/10) — покращена версія починається "Ellanse не тільки забезпечує..." — це знову монолог менеджера. Тренер хоче бачити покращену версію ЯК ДІАЛОГ:
```
Менеджер: «Питання...»
Лікар: [можлива відповідь]
Менеджер: «Факт + закриваюче питання»
```

**Рішення:** В EVALUATE змінити формат покращеної версії:
```
💬 *Покращена версія (діалог):*
М: «[питання]»
→ Л: [можлива реакція]
М: «[факт + next step]»
```

## 3. АРХІТЕКТОР ЗАПИТІВ

### Діагноз: Routing працює, але classification — грубий

**Проблема 8: Бінарна класифікація замість спектру**
Зараз: є "заперечення" keyword → SOS. Але заперечення бувають:
- **Холодне** ("не потрібно, вже є постачальник") — потрібен інший підхід ніж для "дорого"
- **Тепле** ("цікаво, але дорого") — менеджер вже на 70% шляху, потрібен один push
- **Конкурентне** ("у нас Sculptra працює") — потрібне порівняння, не перевага
- **Емоційне** ("не вірю в нові продукти") — потрібна емпатія, не факти

Бот дає ОДНАКОВИЙ SOS на всі 4 типи.

**Рішення:** Розширити EXTRACT щоб визначав "температуру" заперечення:
```
ТЕМПЕРАТУРА ЗАПЕРЕЧЕННЯ:
- ХОЛОДНЕ (лікар не зацікавлений зовсім) → стратегія: створити потребу через SPIN
- ТЕПЛЕ (зацікавлений, але є бар'єр) → стратегія: прибрати бар'єр фактом
- КОНКУРЕНТНЕ (лоялен до іншого бренду) → стратегія: диференціація, не критика
- ЕМОЦІЙНЕ (страх/недовіра) → стратегія: емпатія + клінічний сценарій
```
І передавати цю температуру в SOS-промпт як інструкцію для стратегії.

**Проблема 9: Type B детекція занадто ключова**
"Я відповіла" → Type B (оцінка). Але менеджер може сказати "я відповіла йому що Ellanse кращий, але він не погодився — що робити?" — це НЕ запит на оцінку, це запит на SOS з контекстом.

**Рішення:** Додати другий рівень перевірки:
```python
# Якщо є і "я відповіла" І "що робити/як/що далі" → це SOS з контекстом, не evaluate
_is_type_b = _has_type_b_kw and not any(kw in t_lower for kw in ["що робити", "что делать", "як далі", "що далі", "як відповісти", "что ответить"])
```

**Проблема 10: Follow-up не передає КОНТЕКСТ попереднього типу**
Якщо менеджер отримав SOS, а потім пише "а якщо він скаже що подумає?" — бот повинен дати follow-up В ФОРМАТІ SOS (з killer phrase, next step). Зараз follow-up йде в INFO або generic coach.

**Рішення:** Зберігати `_last_coach_type` в state і використовувати для follow-up routing:
```python
if _is_coach_followup and chat_history:
    _last_type = state_data.get("last_coach_type", "info")
    # Продовжуємо в тому самому форматі
```

## 4. ДИРЕКТОР З ПРОДАЖІВ

### Діагноз: Бот корисний, але не стає НЕЗАМІННИМ

**Проблема 11: Відсутня персоналізація під конкретного лікаря**
Менеджер пише "лікар Іванова з клініки Esthetic каже що дорого" — бот ігнорує "Іванова" і "Esthetic". Якщо б бот запам'ятовував лікарів (хоча б в рамках сесії) — менеджер відчував би що бот ЗНАЄ його клієнтів.

**Рішення (мінімальне, без БД):** Витягувати ім'я лікаря і клініку з тексту і включати в відповідь:
```
⚡ SOS для *лікаря Іванової* (клініка Esthetic):
```
Це мінімальна зміна (в llm_user_text), але створює відчуття персоналізації.

**Проблема 12: Немає "що НЕ говорити"**
Тести показують що бот дає аргументи ЗА, але ніколи не каже що НЕ РОБИТИ. Досвідчений тренер завжди каже: "І ось чого НІКОЛИ не кажи лікарю в цій ситуації: [...]"

**Рішення:** Додати в SOS секцію:
```
⛔ *Чого НЕ казати цьому лікарю:*
- [фраза яка точно відштовхне — і чому]
```

**Проблема 13: VISIT — немає "план B"**
Тест VISIT показує гарний брифінг, але немає плану на випадок якщо лікар взагалі не зацікавлений. "Що робити якщо через 5 хвилин зрозумієш що лікар не хоче розмовляти?"

**Рішення:** Додати в VISIT:
```
🆘 *План B (якщо лікар не зацікавлений):*
- Не тисни. Залиш зразок/каталог + візитку
- Фраза на вихід: «[конкретне речення для graceful exit]»
- Наступний крок: повернися через [час] з конкретною пропозицією
```

## Позиція кожного експерта

### Промпт-інженер:
> Архітектура v2 правильна — модульність працює, extract працює. Але GPT досі ФОРМАТУЄ а не ДУМАЄ. Головна причина — промпт показує ВСІ варіанти (всі гілки, весь SPIN) і GPT намагається заповнити ВСЕ замість того щоб ОБРАТИ найкраще. Рішення: менше шаблону, більше decision-making інструкцій. Також критично — few-shot для killer phrase, бо зараз GPT не розуміє що це.

### Тренер з продажів:
> Бот покращився з 4.5/10 до 6/10. Evaluate і Feedback — революція, раніше цього не було. Але SOS досі = "розумний шаблон" а не "коучинг". Коуч адаптується під ситуацію, а бот дає однакову структуру. Ключове: антимонолог + покращена версія як діалог. І обов'язково "чого НЕ казати" — це те що відрізняє тренера від ChatGPT.

### Архітектор запитів:
> Routing v2 кращий за v1 на порядок. Але classification занадто грубий — 5 типів замість спектру. Температура заперечення — це те що дасть реальний стрибок якості з мінімальним кодом (зміна тільки в EXTRACT). Type B detection потрібно уточнити — "я відповіла + що робити" це SOS а не evaluate. І обов'язково зберігати last_coach_type для follow-up.

### Директор з продажів:
> Бот став КОРИСНИМ. Evaluate (3/10 за "дешевше") — саме те що потрібно, менеджери бачать свої помилки. Visit — нарешті! Менеджер може підготуватися за 2 хвилини. Але бот досі безликий — не знає лікаря, не каже чого уникати, не має плану B. Хочу щоб менеджер відчував що бот ЗНАЄ його роботу, а не просто відповідає на запити.
