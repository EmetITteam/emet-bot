# Scope C — на подумать (відкладено)

Вирішено 23.04.2026 не робити одразу. Повертатись коли:
- Стане більше 20 активних менеджерів (зараз 9) → потреба в UX-оптимізаціях зросте
- Менеджери почнуть просити швидше (фідбек "довго друкувати")
- Junior'и почнуть лажати на промптах senior-рівня (видно через quality_history по level)

---

## C-1. Inline keyboard у Coach-режимі

### Що це
При вході в Coach-режим (`/coach` або кнопка) Telegram показує inline-клавіатуру з кнопками типових сценаріїв:

```
┌─ 💼 Sales Coach ─────────────────┐
│  💰 Дорого                       │
│  🤝 Працюю з конкурентом         │
│  ❓ Не довіряю                   │
│  ⚠️ Побічка / реакція            │
│  📊 Порівняння препаратів        │
│  📋 Протокол моно                │
│  🔗 Combo                        │
│  ✏️  Свій запит                  │
└──────────────────────────────────┘
```

Клік на кнопку → бот:
1. Запам'ятовує тип сценарію в FSM
2. Просить ввести конкретику ("Який препарат? Яке заперечення лікаря дослівно?")
3. Класифікатор отримує підказку intent → краще працює

### Навіщо
- **Швидкість для менеджера:** 1 клік замість 10 секунд набору формулювання
- **Якість для бота:** intent заздалегідь відомий → менше помилок класифікатора
- **Навчальний ефект:** менеджер бачить які сценарії бот вміє → не питає те що не вміє

### Як технічно
- aiogram `InlineKeyboardMarkup` + `InlineKeyboardButton(callback_data="scenario:price")`
- Новий FSM state `UserState.coach_scenario_picked`
- Хендлер `@dp.callback_query(F.data.startswith("scenario:"))` зберігає тип у state і просить уточнення
- Класифікатор отримує `prefilled_intent` через параметр і не калькулює з нуля
- Час: ~2-3 години

### Ризики
- Менеджери звикнуть до кнопок і **не задаватимуть** свої запити (звуження сценаріїв)
- Кнопки "застаріють" — зміниться методика продажу, треба буде апдейтити
- При >7 кнопках Telegram погано рендерить на мобільних
- Може бути зайвим тертя якщо менеджер вже знає що писати

### Коли повертатись
- Коли Юлія/Александра попросять "хочу швидше"
- Коли появляться нові типи заперечень — кнопки можна додавати поступово
- Коли quality_history покаже що 60%+ запитів попадають в 5 категорій → варто кнопки

### Альтернатива (простіше)
Slash-команди: `/dorogo`, `/komp`, `/pobichka` — менеджер сам набирає, бот одразу знає intent. **Час:** 1 година. Менш візуально, але без ризику обмеження.

## C-2. Junior vs Senior сегментація промптів

### Що це
У таблиці `users` вже є поле `level` (1-5+). Зараз воно ніде не використовується для промптів. Ідея — генерувати **різні відповіді в залежності від рівня**:

| Аспект | Junior (level 1-2) | Senior (level 3+) |
|---|---|---|
| Тон | пояснювальний, як коуч | професійний, як партнер |
| Деталі | базові терміни розшифровуються | передбачає що менеджер знає термін |
| Формат | широкі секції, багато прикладів | стислі тези, edge cases |
| Cross-references | "детальніше у курсі LMS X" | без LMS-посилань (вже пройшов) |
| Killer phrase | повна довга фраза | коротка для досвідченого |
| Приклади | 2-3 типових | 1 нестандартний |

### Навіщо
- **Junior:** не злякати складністю → швидше адаптується (зараз бот пише як для senior, новачку важко)
- **Senior:** не "розжовувати очевидне" → бот не сприймається як "дитячий" (зараз скаржаться на простирадла тексту)
- **Для бізнесу:** одразу видно через quality_history чи різниця в helpfulness між рівнями зменшується

### Як технічно
- В `main.py` після `get_user_role(user_id)` отримати `level` з БД
- Передати `{user_level}` як параметр у prompt assembly
- В `PROMPT_COACH_BASE` додати умовну секцію:
  ```
  ## Рівень менеджера: {user_level}
  - level=1-2 (junior): {junior_instructions}
  - level=3+ (senior): {senior_instructions}
  ```
- АБО краще — два окремих base prompt: `PROMPT_COACH_BASE_JUNIOR` / `PROMPT_COACH_BASE_SENIOR`
- Оновити LLM-judge щоб оцінював відповідність рівню
- Час: 2-3 години + 1 година на тестування на реальних менеджерах

### Ризики
- **Хтось образиться:** Junior подумає "мене за тупого тримають", Senior — "мене позбавили деталей"
- **Класифікація рівнів суб'єктивна:** хто визначає коли level переходить з 2 на 3?
- **Поточні `level` у БД можуть бути неправильні:** треба audit + recalibration перед roll-out
- **Двократне зростання promt-розміру:** дві версії коштують token'и + складніше підтримувати

### Коли повертатись
- Коли менеджерів стане >15 і явно проявляться 2 групи "хто хоче коротше" і "хто хоче пояснень"
- Коли LLM-judge стабільно покаже що role_awareness низький саме для junior'ів (можна зчитувати з quality_history per-user)
- Після роботи з HR — щоб level в БД відповідав реальному рівню менеджера

### Альтернатива (простіше)
**User preference** замість автоматичної сегментації — у кожного user поле `preference: short|detailed`, менеджер сам обирає в /settings. Менш magic, більше контролю користувача. **Час:** 1.5 години.

## Загальна примітка

Обидва скоупи — про **UX оптимізацію**. Технічно нескладно зробити (5-7 годин разом). Але обидва несуть ризик "перетравливания" — додаємо контролі/опції/умови, які потім доводиться підтримувати. Краще робити коли є чіткий тригер у даних або у фідбеку, а не "бо здається корисно".

Якщо повертатись — спочатку подивитися:
1. Останні 200 діалогів у `logs` — скільки унікальних форматів запитів? Якщо <8 категорій покривають 80% — кнопки точно допоможуть.
2. `quality_history` за 30 днів per-user (треба дописати) — чи є кореляція рівень↔helpfulness? Якщо так — сегментація.
3. Прямий фідбек від Юлії: "що мене найбільше бісить у боті за останній тиждень". Якщо в топ-3 згадуються "довго писати" або "багато тексту" — пріоритет UX.
