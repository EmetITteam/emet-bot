# Рекомендації для ефективної роботи з AI

## 1. Що вивчити

### Пріоритетне (дасть найбільший ефект):
| Тема | Чому | Час | Ресурс |
|------|------|-----|--------|
| **Основи Git** | branching, diff, stash — точніші інструкції AI | 4-5 год | GitHub Skills (безкоштовно) |
| **Docker basics** | volumes, compose, logs — самостійна діагностика | 3-4 год | Docker Getting Started |
| **SQL запити** | SELECT, JOIN, WHERE — перегляд БД без AI | 3-4 год | SQLBolt (безкоштовно) |
| **Промпт-інженерінг** | структура промптів, chain-of-thought, few-shot | 6-8 год | Anthropic Prompt Engineering docs |
| **RAG концепції** | Embeddings, chunking, retrieval — діагностика RAG | 2-3 год | LangChain docs, розділ RAG |

### НЕ варто вчити (AI зробить краще):
- Синтаксис Python в деталях
- CSS/HTML верстка
- Конфігурація nginx/DevOps деталі
- Алгоритми і структури даних

## 3. Шаблони запитів до AI

### Баг-репорт:
```
[скріншот]
Проблема: бот каже X, а правильно Y.
Джерело правильних даних: [документ / курс / знання].
```

### Нова фіча:
```
Потрібно: [хто і що робить зараз — pain point]
Хочу: [бажаний результат]
Формат/обмеження: [файли, інтерфейс, хто користується]
Хто користується: [імена, рівень технічності]
```

### Системна проблема:
```
За останній тиждень N скарг від менеджерів:
1. [скрін] — опис проблеми
2. [скрін] — опис проблеми
Бачу патерн — може проблема в [RAG / промпті / даних]?
```

### Початок нової сесії:
```
Контекст: [назва проекту, домен].
Прочитай ARCHITECTURE.md для розуміння проекту.
Задача сьогодні: [конкретно].
Обмеження: [дедлайн / не чіпати X / тільки Y].
```

### Перед складною задачею:
```
Перед тим як робити — скажи план.
Що може піти не так?
Зроби тест перед деплоєм.
```

## 4. Workflow

### Базовий (робочий):
```
Ідея → Claude пише → тест в боті → скріншот помилки → Claude фіксить
```

### Покращений:
```
Ідея → Опис з контекстом → Claude пише план → Підтвердження →
Claude пише + авто-тестує → Деплой → Ручний тест →
Фідбек від користувачів → Quality Monitor ловить автоматично →
Нова ітерація
```

## 5. Правила взаємодії з AI

### Що працює (продовжувати):
- ✅ Скріншоти з реального бота — найцінніший input
- ✅ "Неправильно, має бути так" з правильною відповіддю
- ✅ "А де ще може бути така проблема?" — системне мислення
- ✅ Делегування агентам паралельних задач
- ✅ Збір фідбеку від кінцевих користувачів (Марина)

### Що додати:
- 📋 "Перед тим як робити — скажи план" (для складних задач)
- ⚠️ "Що може піти не так?" (після великих змін)
- 🧪 "Напиши тест, прогони, покажи результат, потім деплой"
- 📄 CLAUDE.md файл з правилами проекту для кожної нової сесії
- 📊 Завжди починати з даних: "вигрузи діалоги" перед оптимізацією

### Чого уникати:
- ❌ Не давати задачу без контексту ("зроби краще")
- ❌ Не правити код вручну без повідомлення AI (десинхронізація)
- ❌ Не пропускати етап тестування перед деплоєм
- ❌ Не намагатись контролювати кожен рядок — перевіряти результат

## 6. Головне правило

**Ваша суперсила — знання бізнесу і кінцевого користувача.**
AI пише код. Ви знаєте що -20°С неправильна температура для Exoxe.
Менеджер з доменним знанням + AI = ефективніше за команду з 3 розробників.

**Фокус:**
1. Збір фідбеку від команди (скріншоти, скарги, побажання)
2. Якість контенту (правильні дані в курсах > правильний код)
3. Стратегічні рішення (розділити індекси, змінити архітектуру)

## 7. Best practices з інших проектів

Аналіз 12 проектів у папці Код/ показав що найкращі практики вже використовуються в emet-legal-check та ПЛанирование. Що запозичити:

### З emet-legal-check (оцінка 8/10):
- **PROJECT_PLAN.md** з фазами, строками, критичним шляхом
- **TZ (технічне завдання)** — 42KB документ з бізнес-контекстом
- **Структура src/** — компоненти розділені по функціях (ui/, checks/, dashboard/)
- **Supabase migrations/** — reproducible DB (зараз у aesthetic_bot схема тільки в коді)

- **ARCHITECTURE.md з діаграмою систем** — ASCII або Mermaid
- **IMPLEMENTATION_PLAN.md** — спринти з залежностями
- **Розділ "Що НЕ реалізовано"** — блокери, обмеження
- **Обгрунтування технічних рішень** — чому обрали X замість Y
